# Push Notification Prototype API Documentation

This document outlines all currently available API endpoints in the backend architecture, replicating the Swagger (`/docs`) specification natively.

**Base URL:** `http://127.0.0.1:8000`

---

## 1. System

### `GET /health`

Basic endpoint to verify CORS mappings, internet stability, and internal router connectivity dynamically.

- **Parameters:** None
- **Responses:**
  - **200 OK**
    ```json
    {
      "status": "ok",
      "message": "API is online"
    }
    ```

---

## 2. Tokens (FCM Registry)

### `POST /tokens/register`

Strictly maps user devices to the backend FCM registry. Handles unique Pydantic validation forbidding injection payloads securely.

- **Request Body** (application/json)
  - `owner_id` (string, required): Internal BSON ID of the targeted staff or doctor.
  - `owner_type` (string, required): Roles securely partitioned for payload routing. (Valid options: `"doctor"`, `"staff"`).
  - `fcm_token` (string, required): Firebase Cloud Messaging device push token natively generated by Mobile/Web apps (min-length: 5).

- **Responses:**
  - **200 OK**
    ```json
    {
      "status": "success",
      "message": "Device token securely mapped and activated."
    }
    ```
  - **422 Validation Error**: Thrown strictly when missing fields or adding illegal Pydantic structures.

---

## 3. Appointments (Booking Control)

### `POST /appointments/request`

Accepts frontend payload structures mapped rigidly to Pydantic constraints. Unconditionally forces a `pending` status natively and dynamically dispatches background requests alerting Staff to the pending booking organically.

- **Request Body** (application/json)
  - `patient_id` (string, required): MongoDB ID of the patient.
  - `doctor_id` (string, required): MongoDB ID of the targeted doctor.
  - `appointment_time` (datetime, required): Formatted ISO 8601 UTC date block (e.g., `2026-04-01T09:00:00Z`).

- **Responses:**
  - **201 Created**
    ```json
    {
      "status": "success",
      "appointment_id": "<generated_bson_id>",
      "message": "Appointment requested."
    }
    ```
  - **404 Not Found**: Triggered strictly if `patient_id` or `doctor_id` do not exist within the MongoDB collections.

### `POST /appointments/{appointment_id}/approve`

Triggers internal decoupled proxies algorithmically flipping the status to `approved`. Safely fires mocked SMS routines sequentially and dispatches BSON decoupled Firebase payloads flawlessly.

- **Path Parameters:**
  - `appointment_id` (string, required): The target appointment's generated ID.

- **Responses:**
  - **200 OK**
    ```json
    {
      "status": "success",
      "message": "Appointment intelligently approved and push payload distributed."
    }
    ```
  - **400 Bad Request**: If the appointment is already natively `"approved"`.
  - **404 Not Found**: If the explicit `appointment_id` is missing.

### `GET /appointments/`

Retrieves paginated appointment arrays securely caching infinite data loads natively.

- **Query Parameters:**
  - `status` (string, optional): Filter purely by status natively (e.g., `"pending"`, `"approved"`).
  - `limit` (integer, optional): Memory constraints blocking excessive load loops (Default: 20).
  - `skip` (integer, optional): Offset scale mapping organically (Default: 0).

- **Responses:**
  - **200 OK**: JSON Array of internal Appointment dictionary objects efficiently mapped.

## 4. Notifications (Inbox Analytics)

### `GET /notifications/`

Retrieves dynamic paginated Notification architectures explicitly targeting a specific user identifier, heavily mapped by timestamp descending cleanly.

- **Query Parameters:**
  - `user_id` (string, required): The core ID of the user natively fetching the data.
  - `limit` (integer, optional): Bounded caching thresholds (Default: 20).
  - `skip` (integer, optional): Pagination cursors indexing organically (Default: 0).

- **Responses:**
  - **200 OK**: JSON Array natively returning all valid `NotificationInbox` architectures.

---

## 5. Doctor Roster Overview

### `GET /doctors/`

Retrieves indexed arrays of all active Doctors decoupling the frontend UI logic from strictly hardcoding Doctor strings globally.

- **Query Parameters:**
  - `limit` (integer, optional): Strict pagination memory barriers natively bounding outputs (Default: 20).
  - `skip` (integer, optional): Offset logic mapping perfectly. (Default: 0).

- **Responses:**
  - **200 OK**: JSON Array of native MongoDB Doctor dictionaries stripped to standard data maps.
